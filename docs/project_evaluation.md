# go-state-transition プロジェクト評価

## 評価概要

このプロジェクトは、Go で状態遷移を扱う最小構成のサンプルから一歩進み、MySQL + GORM を使った取得処理まで含めて整理されたサンプルになっています。  
特に、`BankStatus` による型付け、`usecase` での遷移制御、`persistence` 層での DB レコードから中間データを経由した変換が揃ったことで、以前より設計意図が明確になりました。

一方で、現時点でも構成は学習用としてかなり小さく、実務向けのサンプルとしては責務の寄せ方や API 設計、エラー表現に改善余地があります。  
総じて、このプロジェクトは「DB の数値ステータスをドメインに安全に持ち込む方法」を示す教材としては十分に良い出来です。

## 良い点

### 1. ステータス表現が明確

`internal/domain/bank_status.go` で `BankStatus` を独自型にしており、DB の `TINYINT` をそのまま業務ロジックへ流していません。  
`Label`、`IsValid`、`CanTransitionTo`、`NewBankStatus` が一箇所にまとまっているため、状態の意味と検証ルールを把握しやすいです。

### 2. 遷移ルールの集中管理ができている

`allowedTransitions` に遷移可能ルートが集約されているため、ユースケースごとに条件分岐が散らばっていません。  
状態遷移のルール変更が発生した場合も、変更点を追いやすい構造です。

### 3. 永続化層で直接ドメインへマッピングしていない

今回追加された `infrastructure/persistence/management_mapper.go` により、`managementRecord` から直接 `domain.Management` を組み立てず、`managementData` を経由して変換しています。  
この構成は、DB スキーマとドメインモデルを密結合させないという点で妥当です。

### 4. 永続化層の責務が比較的整理されている

`management_repository.go` は DB 取得と保存を担当し、ステータスの意味づけそのものは `domain.NewBankStatus` に委ねています。  
「DB の値を読み出す責務」と「値の妥当性をドメインで判定する責務」が分かれており、サンプルとして筋が良いです。

### 5. テストが層ごとに揃っている

以下の観点が分かれてテストされている点は良いです。

- `internal/domain/*_test.go`: ドメインルール
- `usecase/management_usecase_test.go`: ユースケースの遷移制御
- `infrastructure/persistence/*_test.go`: DB 取得と変換

小規模でもレイヤごとの確認ポイントが見えるので、学習用として読みやすいです。

### 6. 実DB を使った取得検証ができる

`docker-compose.yml` と `testhelper/db.go` があり、MySQL を起動して統合テストできるため、単なるインメモリ例では終わっていません。  
「DB の数値をどう扱うか」という今回の主題に対して、検証方法も揃っています。

## 気になる点

### 1. README の構成説明が現状に追いついていない

README には `infrastructure/persistence` や MySQL 関連の説明がまだ反映されていません。  
現在の実装と README の差分があるため、初見ではプロジェクトの全体像を誤解しやすいです。

### 2. `Management` エンティティはまだ薄い

`internal/domain/management.go` は生成関数が追加されたものの、エンティティとしての振る舞いはほぼありません。  
実質的な業務ルールは `BankStatus` と `usecase` に載っており、`Management` 自体が主役になり切れていません。

### 3. ユースケースの実装が重複している

`StartProcessing`、`Complete`、`Suspend` はほぼ同じ処理の繰り返しです。  
サンプルとしては明快ですが、状態が増えるとメソッド数と重複が増えやすい設計です。

### 4. 保存時の防御が弱い

`Save` は `domain.Management` をそのまま更新していますが、`RowsAffected` の確認や、存在しない ID 更新時の明確なエラー判定はありません。  
実務コードとして見ると、更新失敗の検知はもう少し厳密にした方がよいです。

### 5. `AllowedNextStatuses` が内部スライスを返している

`allowedTransitions` の値をそのまま返しているため、呼び出し側で変更できてしまいます。  
今のコード規模では問題化しにくいですが、不変条件を強くしたいならコピーを返す方が安全です。

### 6. エラーの扱いが文字列中心

遷移不可や未存在、ステータス不正がすべて文字列ベースの `error` です。  
教材としては十分ですが、呼び出し側でエラー種別を判定したい場面を考えると、専用エラーの方が拡張しやすいです。

## 総合評価

- 学習用サンプルとして: 良い
- 状態遷移と DB 取得の橋渡し例として: 良い
- 実務設計の完成例として: まだ小さい
- レイヤ分離の見本として: 十分に参考になる

以前は「状態遷移の概念サンプル」に寄っていましたが、今は「状態遷移 + 永続化変換の基本例」として評価できます。  
特に、「DB から取った値をいきなりドメインへ流し込まない」という要件に対して、構造で答えられている点はこのプロジェクトの強みです。

## 改善提案

### 1. README を現在の構成に合わせて更新する

少なくとも以下は README に書くべきです。

- `infrastructure/persistence` の役割
- `managementRecord -> managementData -> domain.Management` の変換意図
- MySQL テストの実行方法

### 2. エンティティへ遷移操作を寄せる

例えば `Management.StartProcessing()` のようにエンティティへ振る舞いを持たせると、ユースケースは取得と保存に集中できます。  
状態遷移の責務をよりドメイン側へ寄せられます。

### 3. リポジトリの保存処理を厳密にする

`Save` では以下を検討する価値があります。

- 更新対象が存在しない場合のエラー化
- 保存前の値検証
- 更新件数の確認

### 4. エラー種別を明示する

`ErrManagementNotFound`、`ErrInvalidBankStatus`、`ErrInvalidTransition` のような形にすると、呼び出し側の分岐が安定します。  
テストも文字列比較に依存しにくくなります。

### 5. 取得以外の永続化パターンも追加する

今回は `FindByID` と `Save` が中心ですが、実務に寄せるなら以下を追加すると一段分かりやすくなります。

- 新規作成
- 一覧取得
- 検索条件付き取得
- トランザクションを伴う更新

## 結論

このプロジェクトは、Go における状態遷移設計の入門用サンプルとして良くできています。  
さらに、永続化層を追加したことで、単なる概念説明ではなく「DB の数値をどう安全にドメインへ持ち込むか」まで示せるようになりました。

今の段階では、実務アプリの完成例というより、設計の基本線をきれいに切り出した教材です。  
README 更新とドメイン責務の強化が入れば、かなり説得力のあるサンプルになります。
